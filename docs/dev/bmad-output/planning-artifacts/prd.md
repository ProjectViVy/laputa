# Product Requirements Document - 天空之城

**Author:** 大湿
**Date:** 2026-04-13

## Executive Summary

天空之城是一个面向长期连续性、主体延续与通用知识沉淀的万能记忆库。产品目标不是做一个更复杂的聊天记录检索器，而是构建一个能够像人一样进行筛选、强化、遗忘、整理与唤醒的统一记忆系统。

天空之城以纯 Rust 实现，并以 `mempalace-rs` 作为首要代码与产品基线。首发阶段不另起炉灶重写底座，而是在 `mempalace-rs` 已验证的本地优先、离线优先、结构化检索、知识图谱、分层记忆与 MCP 工具化接口之上继续演化；同时适度移植 `agent-diva` 的 memory domain / diary / identity 思路，吸收 `UPSP` 的主体协议理念与 `LifeBook` 的时间流压缩思想，把这些来源统一收口为一个新的长期记忆产品。

天空之城的差异化不在于“能不能搜到旧内容”，而在于三点：
- 以**时间流记忆**为主轴，而不是以语义检索为主轴。
- 以**主体连续性**为目标，而不是只为单轮回答补上下文。
- 以**筛选、情绪、热度、节律整理、遗忘与归档**构成完整生命周期，而不是只做存储与召回。

目标用户是需要长期、可控、私有化记忆的个人 AI 使用者、伴侣型 AI 用户、研究者、长期项目协作者与 AI 原生开发者。产品策略不是删除原始设计中的远期能力，而是按阶段实现：第一阶段先交付最小闭环——日记输入、节律整理、周级胶囊、唤醒包注入与按需检索；后续阶段再逐步补齐关系、情绪、归档、多周期压缩与更复杂的动态编排能力，避免在第一阶段扩张为多宿主、多身份、多后端的大而全系统。

## Success Criteria

1. 用户可以在本地、离线环境中创建并持续维护一个可运行的长期记忆库，无需依赖云端记忆服务。
2. 系统在单次会话启动时能生成受控大小的唤醒上下文，默认不超过 1200 tokens，并覆盖身份、近期阶段、关键关系与最近节律摘要。
3. 用户写入日记后，系统可以在节律点自动生成至少一层摘要胶囊（周级），并可用于后续唤醒或回忆。
4. 系统可以区分至少四类核心记忆对象：身份/主体、事件/日记、关系、摘要胶囊，并支持统一读取与查询。
5. 用户可以显式标记“记住”“忘掉”“重要”“情绪锚定”等干预动作，并使其影响后续热度、保留与归档结果。
6. 系统可在普通模式下自动衰减低价值内容，并在归档后保留用户考古能力，而不是永久删除。
7. MVP 发布时至少具备一条端到端演示链路：`daily diary -> weekly capsule -> wakeup context -> contextual answer`。
8. 全部核心运行链路使用纯 Rust 实现，且可在无 Python 运行时依赖的环境中完成安装、初始化、写入、整理、检索与导出。

## Product Scope

### Product Baseline

天空之城的明确产品基线如下：

1. **代码基线**：以 `mempalace-rs` 为主底座继续开发，而不是从零重写新的记忆引擎。
2. **能力主轴**：以时间流记忆为主轴，以语义检索为辅，不把产品定义成普通 RAG 工具。
3. **主体目标**：强调主体连续性、关系连续性与长期状态延续，而不是只服务单轮回答补上下文。
4. **理念来源**：
   - `mempalace-rs` 提供结构化存储、检索、wakeup、CLI/MCP 与记忆栈底座。
   - `LifeBook` 提供日→周→月→季→年的时间流压缩与滚动回看理念。
   - `UPSP` 提供 identity / state / relation / STM / LTM 的主体协议视角。
   - `agent-diva` 提供 memory domain、diary 分区、identity / soul 延展方向的参考。
5. **开发策略**：先在 `mempalace-rs` 之上新增“时间流、主体协议、节律整理、双轨记忆”等新概念，再视需要适度移植 `agent-diva` 的局部能力，不以完整兼容全部前置项目为首发目标。

### MVP Scope

MVP 是第一阶段交付，不代表产品最终能力边界。MVP 只包含以下能力：

1. **主体初始化**
   - 创建基础身份定义。
   - 创建状态、关系、记忆与节律所需的最小数据结构。
   - 初始化可被 wakeup 与 recall 使用的最小关系字段。

2. **日记输入层**
   - 支持写入 daily diary。
   - 支持理性/感性双分区的轻量标记，或等价元数据标记。
   - 支持最小标签与来源字段。

3. **节律整理层**
   - 支持按节律点触发整理。
   - MVP 至少实现周级摘要胶囊。
   - 生成适合唤醒与检索的结构化 capsule。

4. **唤醒上下文层**
   - 输出受控长度的 wakeup context。
   - 默认包含：身份、近期状态、最近周期摘要、关键关系、关键未完成事项。

5. **检索层**
   - 支持按时间流优先的 recall。
   - 支持按需语义检索作为补充路径。
   - 支持统一的 MCP/CLI 调用接口。

6. **生命周期治理层**
   - 支持热度、重要性、情绪权重、复习次数等最小保留字段。
   - 支持手动“important / forget / emotion anchor”干预。
   - 支持归档候选与归档导出，但 MVP 不要求完整考古系统 UI。

### Growth Scope

1. 月度、季度、年度压缩链。
2. 更完整的艾宾浩斯反向复习与自动衰减策略。
3. 关系驱动召回与情绪驱动召回。
4. 更强的 knowledge graph 与 Hall/Tunnel 连接机制。
5. 适度引入 `agent-diva` 风格的完整双轨记忆结构，以及更丰富的 identity / soul 相关结构。
6. 多宿主适配：agent runtime、桌面端、CLI、MCP 客户端。
7. 多身份切换与多位格协同。
8. 档案考古工具链：archive unpack、dig、索引与回看视图。

### Vision Scope

1. 成为统一的 AI 长期记忆底座，而不是某个单独产品的插件。
2. 支持个人、伴侣、项目、团队、研究与创作场景下的统一记忆表达。
3. 实现“主体连续性即产品能力”：模型切换、客户端切换、会话断裂后仍保持可审计的记忆连续。
4. 从“记住过去”升级为“基于过去形成稳定自我与长期关系”。

## User Journeys

### Journey 1：个人长期记忆维护

用户每天写下若干日记、事件、想法与情绪片段。系统将其作为原始素材保留，并在节律点自动整理为更短的摘要胶囊。下次唤醒 AI 时，系统优先注入身份、近期状态和最新节律摘要，而不是粗暴回放全部原文。

### Journey 2：伴侣型 AI 连续相处

用户希望 AI 记住关系温度、近期重要事件、共同经历和情绪波动。系统保留连续时间流、关系节点与情绪锚点，在后续对话中能先基于近期时间流回忆，再补充相关语义检索结果，使回应更连贯、更符合关系上下文。

### Journey 3：项目与知识长期沉淀

开发者或研究者使用天空之城保存架构决策、实验记录、阶段性结论与失败原因。系统在长期运行中对这些内容进行筛选、强化、摘要与归档，使后续 AI 能回忆“我们为什么这样做”，而不只返回零碎片段。

### Journey 4：主动遗忘与考古回看

用户不希望系统无限膨胀，也不希望系统私自销毁记忆。系统会将低热度内容标记为归档候选并打包，正常对话不主动索引归档层，但用户可以在需要时手动考古和回看旧时期内容。

### Journey 5：模型或宿主切换

用户更换模型、客户端或运行宿主后，仍可通过同一套结构化记忆与导出机制保留连续性。主体不是绑定在某个模型参数中，而是绑定在天空之城的记忆结构与协议上。

## Domain Requirements

本产品不属于强监管行业，但具备以下领域特性要求：

1. **私有化优先**：默认本地存储，本地可导出，本地可迁移。
2. **离线优先**：核心链路在无外部网络条件下可运行。
3. **数据主权**：用户可审阅、导出、标记与归档自己的全部记忆数据。
4. **可解释性**：系统需要尽可能说明为何保留、强化、遗忘或归档某条记忆。
5. **反脆弱性**：系统不应因依赖复杂外部服务而提高维护门槛。

## Innovation Analysis

天空之城的创新点不在于新增一套向量检索封装，而在于把以下能力收口到一个统一产品定义中：

1. **人脑模拟记忆生命周期**：注意力筛选、情绪标记、重复强化、遗忘曲线、睡眠整理、主动遗忘。
2. **时间流优先**：先回到“最近发生了什么”，再决定是否做深检索。
3. **主体连续性**：身份、关系、状态与记忆不是松散附件，而是同一系统中的一等对象。
4. **双轨内容**：理性记忆与感性记忆既分层又协作。
5. **归档而非删除**：系统对低热度内容进行打包，不替用户做不可逆删除决策。

## Project-Type Requirements

1. 核心运行链路必须为纯 Rust。
2. 必须继承并复用 `mempalace-rs` 已验证的核心能力路线，优先采用“基于现有底座演化”的策略，而不是另起一套完全无关的基础设施。
3. 第一阶段的核心开发应尽量直接建立在 `mempalace-rs` 的现有模块、CLI 命令与数据结构之上。
4. `agent-diva`、`UPSP` 与 `LifeBook` 的价值优先体现在理念、协议与局部能力迁移，不作为首发阶段的并行主底座。
5. 应优先以 CLI 与 MCP 作为第一阶段交付接口。
6. 应支持本地文件、结构化 SQLite 数据与可扩展的检索索引共存。
7. 应允许后续与宿主系统集成，但 MVP 不依赖宿主存在即可独立运行。

## Functional Requirements

### FR-1 主体初始化
用户可以创建一个新的天空之城记忆库，并生成最小可运行的身份、状态与记忆布局。

### FR-2 日记写入
用户可以写入新的日记条目，并指定或推断时间、分区、标签、来源与情绪锚定信息。

### FR-3 记忆筛选
系统可以对新增内容执行可解释的保留判断，包括存储、丢弃、合并或加热既有条目。

### FR-4 统一记忆记录
系统可以将事件、关系、状态、摘要胶囊与主体信号保存为统一结构的记忆记录，并支持稳定 ID。

### FR-5 节律整理
系统可以在节律点对新增素材进行整理，并生成至少周级摘要胶囊。

### FR-6 唤醒包生成
系统可以输出一个用于 AI 启动时注入的唤醒包，默认覆盖身份、近期状态、最近周期摘要、关键关系和关键任务。

### FR-7 时间流召回
用户或上层 agent 可以按时间窗口、周期层级或最近阶段召回相关记忆。

### FR-8 深度检索
用户或上层 agent 可以对记忆库发起按需深检索，以获取与当前问题最相关的历史条目。

### FR-9 关系记忆
系统可以记录并召回人与人、人与项目、人与主体之间的最小关系变化与重要互动，MVP 至少支持关键关系字段与近期关系摘要。

### FR-10 情绪锚定
系统可以记录记忆条目的情绪权重，并使其影响保留、强化或召回顺序。

### FR-11 手动治理
用户可以对记忆条目执行至少以下干预：标记重要、标记遗忘、设置情绪锚点、手动回看。

### FR-12 归档候选
系统可以识别低热度、久未调用内容，并将其标记为归档候选。

### FR-13 归档导出
系统可以将归档候选打包导出，并保留用户后续解包与考古的能力。

### FR-14 数据导出与迁移
用户可以导出其主体定义、摘要胶囊与核心记忆记录，用于迁移到新设备、新模型或新宿主。

### FR-15 工具化接口
系统可以通过 CLI 与 MCP 暴露初始化、写入、整理、唤醒、召回、检索、标记与归档等能力。

## Non-Functional Requirements

### NFR-1 纯 Rust 运行
系统应在无 Python 运行时依赖的前提下完成核心安装、初始化、写入、整理、召回、检索与导出流程。

### NFR-2 离线可用性
系统应在无外部网络条件下完成全部 MVP 核心能力，且本地数据读写与召回能力不依赖远程服务。

### NFR-3 唤醒成本控制
默认唤醒包应控制在 1200 tokens 以内，且 95% 的普通唤醒请求不需要全库扫描。

### NFR-4 本地响应性能
在 100,000 条以内记忆记录规模下，普通唤醒请求应在 500ms 内返回，深检索请求应在 2s 内返回首批结果。

### NFR-5 数据可解释性
对任一条被保留、强化、遗忘或归档的记录，系统应保留可审查的原因字段或等价解释元数据。

### NFR-6 可迁移性
用户可通过文档化格式导出核心主体、关系、摘要与记忆数据，并在新实例中恢复，不依赖专有云服务。

### NFR-7 稳定性
核心记忆存储需支持异常恢复，单次异常退出后不得造成已确认写入记录的结构性损坏。

### NFR-8 扩展性
MVP 的数据模型应允许后续增加月/季/年节律、更多主体分区与更丰富归档层，而无需重写全部基础结构。

### NFR-9 隐私默认值
系统默认本地保存，不主动上传、不默认同步、不默认将原始记忆内容发送至第三方服务。

### NFR-10 可测试性
MVP 每一条核心链路均应具备自动化测试：初始化、写入、节律整理、唤醒包生成、召回、检索、归档导出。

## Explicit Non-Goals

以下内容不属于 MVP：

1. 多身份并行协同与复杂多位格调度。
2. 图形化桌面端或完整可视化界面。
3. 复杂的月/季/年全量自动压缩链。
4. 归档考古的完整交互式浏览器。
5. 多设备自动同步服务。
6. 面向团队 SaaS 的云托管版本。

## Open Questions (已解决)

以下问题已在 2026-04-13 决策会议中解决：

### Q1: agent-diva 的首个嵌入路径

**决策**: C - Phase 1 仅 diary 分区概念。

**理由**: 天空之城是为 agent-diva 记忆模块开发，但必须具备高度可移植性——可在 openfang 等 Rust agent 项目中使用，未来还有 Python API。

### Q2: 主体定义结构

**决策**: 不保留 UPSP 七文件结构，只取**共振度（关系节点）这一个概念**。

**理由**: UPSP 的复杂概念（工化指数、动态六轴、变速轮等）不适合天空之城。大湿明确不喜欢工化指数等概念。

**明确排除**:
- 工化指数 (workhood_index)
- 动态六轴中的 focus/mood/humor/safety（UPSP自创，学术依据不强）
- 核心六轴
- 变速轮 (speed_wheel)
- 模型戳
- 七文件结构
- 权重形态 [F]/[S]/[A]

**保留三个概念**:
- 共振度 (Resonance) —— 用于关系节点，-100~+100 范围
- valence —— 情感效价（-100~+100），用于情绪胶囊、共振度更新
- arousal —— 激活程度（0~100），用于情绪胶囊可视化

### Q3: 语义检索/RAG

**决策**: RAG 在天空之城设计中不可或缺，MVP 必须启用。

**理由**: LifeBook 反 RAG 的观点不代表天空之城。语义检索与时间流检索是互补关系，不是对立关系。

### Q4: 归档导出物理格式

**决策**: 沿用 mempalace-rs 已有的结构。

**理由**: 保持兼容性，不重新定义天空之城专用打包格式。

### 额外决策

1. **双轨记忆**: 不再采用。改用"热度+主题"自动归档，比理性/感性更复杂。
2. **项目主目录**: Laputa。强调可移植性强兼容。
3. **AGENTS.md**: 必须创建，作为项目规范。

详细决策记录见: `Laputa/DECISIONS.md`
