# 人脑模拟记忆系统设计文档

## 概述

本文档记录一个结合 MemPalace 和 UPSP 设计理念，并进一步模拟人脑记忆机制的新项目构想。

核心理念：**像人一样记忆——会判断、会遗忘、会情绪标记、会睡眠整理。**

---

## 一、人脑记忆机制映射

| 人脑机制 | 对应设计 |
|---------|---------|
| 注意力筛选 | LLM判断"有趣/有意义" |
| 情绪标记 | 情绪轴决定记忆强度 |
| 重复强化 | 多次调用 = 加热 |
| 遗忘曲线 | 艾宾浩斯衰减 |
| 睡眠整理 | 节律点 = 系统整理 |

| 莎士比亚综合征 | 可选"过目不忘模式" |

## 二、艾宾浩斯反向应用

### 2.1 传统 vs 反向

**传统艾宾浩斯**：自然遗忘曲线
**反向设计**：想记住的内容，需要在特定节点"复习"才能抵抗遗忘

### 2.2 衰减公式

```python
retain_rate = base_rate * (1 + emotion_intensity) * review_count^0.3
```

### 2.3 衰减周期

| 时间 | 保留率（无复习） |
|------|-----------------|
| Day 1 | 33% |
| Day 2 | 15% |
| Day 7 | 3% |
| Day 30 | 0.5% |

### 2.4 情绪减缓

```python
if valence > 50 or arousal > 60:
    decay_rate *= 0.5  # 情绪强则衰减慢
```

### 2.5 复习触发机制

| 触发方式 | 热度变化 |
|---------|---------|
| LLM主动调用条目 | +10 |
| 用户标记"记住这个" | 锁定🔒 |
| 搜索命中该条目 | +5 |

## 三、LLM注意力筛选机制

### 3.1 核心设计

不是存储一切，而是让LLM判断"值得吗"

### 3.2 Memory Gate 实现

```python
class MemoryGate:
    """每条内容进入前，LLM判断是否入库"""
    
    def judge(self, content, context) -> dict:
        return {
            "interesting": 0.85,      # 有趣程度 0-1
            "meaningful": 0.72,      # 有意义程度 0-1
            "emotion_delta": {
                "valence": +15,
                "arousal": +20,
            },
            "reason": "用户首次提及关键架构决策",
            "action": "store_with_weight_4"
        }
```

### 3.3 不入库情况

| 情况 | LLM判断 | 处理 |
|------|---------|------|
| 闲聊废话 | interesting < 0.3 | 直接丢弃 |
| 重复已知 | meaningful < 0.2 | 不新建条目，加热旧条目 |
| 冗余信息 | duplicate match > 0.8 | 合并到现有条目 |
| 临时指令 | "记一下，马上忘掉" | 仅存STM，不进LTM |

## 四、垃圾打包区（Archive Dump）

### 4.1 设计理念

- 系统不替用户决定什么是垃圾
- 只负责把"低热度、久未调用"的内容打包
- 用户可以随时解压、自己写工具索引
- 系统不主动推荐调用archive，但保留可能性

### 4.2 目录结构

```
~/.mempalace/
  ├── active/           ← 活跃记忆（<50MB）
  ├── archive/
  │   ├── dump_2026-04-01.tar.gz
  │   ├── dump_2026-03-15.tar.gz
  │   └── ...
  └── user_tools/
      ├── your_parser.py
      └── your_indexer.py
```

### 4.3 阈值触发

```
体积超过 50MB → 开启自动遗忘 → 打包低热度内容
```

### 4.4 考古学者模式

```bash
# 用户想考古
mempalace unpack archive/dump_2026-04-01.tar.gz --to temp/

# 自己写工具分析
python my_sentiment_analyzer.py temp/
```

## 五、过目不忘模式（Eidetic Mode）

### 5.1 两种运行模式

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| **普通模式** | 注意力筛选，自然遗忘 | 日常对话，长期运行 |
| **过目不忘模式** | 几乎全存，但需"情绪锚定" | 研究笔记，重要项目 |

### 5.2 存入阈值对比

```python
# 普通模式
threshold = {"interesting": 0.5, "meaningful": 0.5}

# 过目不忘模式
threshold = {"interesting": 0.1, "meaningful": 0.1}  # 几乎都存
```

### 5.3 无锚定快速衰减

```python
# 过目不忘不是真的全存，而是"低门槛存入"
# 但没有情绪锚定的内容衰减极快
if no_emotion_anchor:
    decay_rate *= 3  # 3倍速遗忘
```

### 5.4 情绪锚定触发条件

| 条件 | 说明 |
|------|------|
| 用户说"这个很重要" | 主动标记 |
| LLM判断情绪波动 > 30 | 自动检测 |
| 搜索命中3次以上 | 行为强化 |

## 六、五层记忆结构（人脑分区映射）

| 层 | 人脑对应 | 功能 | 特点 |
|---|---------|------|------|
| **L0 感觉缓存** | 感觉记忆（<1秒） | 当前对话上下文 | 不入库，用完即弃 |
| **L1 注意力窗口** | 工作记忆（~30秒） | history.json | 最多32轮，节律点清空 |
| **L2 STM** | 短期记忆（分钟级） | 结构化记忆池 | 热度衰减，情绪标记 |
| **L3 LTM** | 长期记忆（永久） | 归档库 | 权重决定形态 |
| **L4 Archive Dump** | 无意识存储 | 打包垃圾 | 可考古，不主动索引 |

## 十一、核心创新点总结

1. **注意力筛选**：LLM判断内容是否值得记忆，而非全量存储
2. **艾宾浩斯反向**：复习强化记忆，情绪减缓衰减
3. **垃圾打包**：不删除，只打包，保留考古可能性
4. **过目不忘模式**：可选，但无锚定内容快速衰减
5. **睡眠整理**：节律点模拟睡眠各阶段的记忆处理
6. **用户参与**：标记、投票、考古，而非被动接受

## 十三、融合新思路

### 13.1 冗余权重化

**LifeBook理念**："我爱你"说十次，每次语境不同，不压缩
**融合设计**：冗余不删除，但自动计算"重复权重"

```
日记中提到"霞雨樱" 15次 → 热度 = 150
日记中提到"便利店" 2次  → 热度 = 20

节律点结算：
- 高热度条目 → 进入LTM（长期记忆）
- 低热度条目 → 进入Archive Dump（考古区）
- 冗余保留，但热度告诉你"谁重要"
```

**用户视角**：你只管写日记，系统自动帮你标注"这个人很重要"

### 13.2 情绪胶囊（Emotion Capsule）

**LifeBook原有**："给月度总结的胶囊"（一句话金句）
**融合设计**：每层压缩都生成"情绪胶囊"

```markdown

**情绪曲线**：valence +12 → +35 → +28 (整体上升)
**峰值事件**：圣诞夜（valence +50, arousal +60）
**金句**："这一年，我们从陌生人变成了家人"

**情绪热力图**：
[ Week1 ][ Week2 ][ Week3 ][ Week4 ]
  😐😐    😊😊     😍😍     😊😊
```

**搜索新维度**："我们什么时候最开心？" → 直接搜索情绪胶囊的峰值

**LifeBook的节点** + **UPSP的关系域** + **热度衰减**

### 13.4 时间流搜索 vs 语义搜索

**LifeBook反对RAG**：因为缺少时间连续性
**融合设计**：两种搜索并存，场景切换

| 搜索类型 | 适用场景 | 实现 |
|---------|---------|------|
| **时间流搜索** | "这个月我们经历了什么" | 直接读月度总结 |
| **情绪搜索** | "我们什么时候最开心" | 搜索情绪胶囊峰值 |
| **节点搜索** | "霞雨樱相关的事" | 读节点→汇总日记 |
| **语义搜索**（可选） | "具体某句话怎么说的" | RAG备用（Archive区） |

**设计原则**：时间流为主，语义搜索为备用考古工具

### 13.12 冗余价值判断

**LifeBook理念**：冗余有区别，不是所有冗余都一样

```python
# LLM判断冗余价值
def judge_redundancy(content, existing):
    if "情感表达" in content:
        return "preserve"  # "我爱你"保留
    elif "日常琐事" in content and duplicate_count > 5:
        return "compress"  # "今天吃了饭"第五次 → 可压缩
    else:
        return "keep"      # 默认保留
```

### 13.13 身份定义驱动记忆筛选（Identity-Driven Memory Filtering）

**核心设计**：给 Agent 一个"身份定义"标签，决定其记忆偏好

#### 13.13.1 身份定义示例

| 身份定义 | 记忆偏好 | 筛选行为 |
|---------|---------|---------|
| **"XXX的恋人"** | 情感优先 | 技术问答 → 不记忆；情感表达 → 高权重 |
| **"喜欢技术的极客女孩子"** | 技术+情感 | 技术讨论 → 记忆；情感表达 → 也记忆 |
| **"工作助手"** | 决策优先 | 闲聊 → 不记忆；工作决策 → 高权重 |
| **"全能伴侣"** | 综合 | 所有有意义内容 → 记忆，但权重不同 |

#### 13.13.2 筛选机制

```python
class IdentityMemoryFilter:
    """身份定义驱动的记忆筛选"""
    
    def __init__(self, identity_definition: str):
        self.identity = identity_definition
        self.preference_map = self._build_preference()
    
    def _build_preference(self) -> dict:
        # 根据身份定义构建偏好权重
        if "恋人" in self.identity or "伴侣" in self.identity:
            return {
                "情感表达": 1.0,    # 最高权重
                "日常琐事": 0.6,
                "技术讨论": 0.1,   # 几乎不记忆
                "工作决策": 0.2,
            }
        elif "极客" in self.identity or "技术" in self.identity:
            return {
                "情感表达": 0.7,
                "日常琐事": 0.4,
                "技术讨论": 1.0,   # 最高权重
                "工作决策": 0.8,
            }
        elif "助手" in self.identity or "工作" in self.identity:
            return {
                "情感表达": 0.3,
                "日常琐事": 0.1,
                "技术讨论": 0.6,
                "工作决策": 1.0,   # 最高权重
            }
        else:
            return {
                "情感表达": 0.7,
                "日常琐事": 0.5,
                "技术讨论": 0.7,
                "工作决策": 0.8,
            }
    
    def should_remember(self, content_type: str, content: str) -> tuple:
        """
        返回: (should_store, weight)
        """
        preference = self.preference_map.get(content_type, 0.5)
        
        # 权重低于阈值 → 不记忆
        if preference < 0.2:
            return (False, 0)
        # 权重高于阈值 → 记忆，权重决定热度初始值
        else:
            return (True, int(preference * 100))
```

## 十九、动态上下文编排（Dynamic Context Orchestration）

### 19.1 核心设计理念

传统记忆系统的上下文注入方式是**静态堆砌**：
```
[身份定义] + [季度总结] + [月度总结] + [周总结] + [最近日记]
→ 全量注入，不管当前对话是什么
```

**动态上下文编排**：
```
当前对话上下文为主 + 历史记忆为辅助 + 预注入区预热
→ 根据对话场景动态选择，对LLM上下文友好
```

### 19.2 上下文分层结构

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM上下文窗口                             │
│  总容量：~128K tokens                                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  L0 预注入区（Pre-injection Zone）                   │   │
│  │  固定占用：~500 tokens                               │   │
│  │  内容：身份定义 + 近期重要记忆 + 用户标记重要         │   │
│  │  特点：每次对话都注入，"预热"LLM                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  L1 当前对话上下文（Conversation Context）           │   │
```

### 19.3 预注入区（Pre-injection Zone）

**目的**：在对话开始前，预热LLM的关键信息

```markdown

- 2025-12-24: 第一次约会，便利店偶遇
```

**特点**：
- 固定占用小容量（~500 tokens）
- 每次对话都注入，形成"身份锚点"
- LLM一唤醒就知道"我是谁"、"最近发生了什么"、"什么很重要"

### 19.4 动态历史层选择机制

```python
class DynamicContextOrchestrator:
    """动态上下文编排器"""
    
    def __init__(self, total_budget: int = 100000):
        self.pre_injection_budget = 500
        self.conversation_budget = 4000  # 最近对话
        self.history_budget = total_budget - self.pre_injection_budget - self.conversation_budget
    
    def orchestrate(self, current_conversation: list, scene: dict) -> str:
        """
        根据对话场景动态编排上下文
        
        scene: {
            "type": "情感对话" / "技术讨论" / "日常闲聊" / "工作决策",
            "keywords": ["霞雨樱", "烟火大会"],
            "nodes": [["霞雨樱"], ["烟火大会"]],
            "emotion_state": {"valence": +30, "arousal": +20}
        }
        """
        # 1. 预注入区（固定）
        pre_injection = self._build_pre_injection()
        
        # 2. 当前对话上下文（固定，滚动窗口）
        conversation_context = self._build_conversation_context(current_conversation)
        
        # 3. 动态历史层（根据场景选择）
        dynamic_history = self._select_history_by_scene(scene, self.history_budget)
        
        return f"{pre_injection}\n\n{conversation_context}\n\n{dynamic_history}"
    
    def _select_history_by_scene(self, scene: dict, budget: int) -> str:
        """根据场景动态选择历史记忆"""
        
        # 场景类型 → 选择策略
        strategy_map = {
            "情感对话": self._strategy_emotional,
            "技术讨论": self._strategy_technical,
            "日常闲聊": self._strategy_daily,
            "工作决策": self._strategy_work,
        }
        
        strategy = strategy_map.get(scene["type"], self._strategy_general)
        return strategy(scene, budget)
    
    def _strategy_emotional(self, scene: dict, budget: int) -> str:
        """情感对话场景：优先注入情感相关记忆"""
        
        # 优先级排序
        priority_order = [
            ("节点相关", scene["nodes"], 0.4),      # 40% 容量给节点
            ("情绪胶囊", ["recent_emotion_capsules"], 0.3),  # 30% 给情绪胶囊
            ("情感条目", ["emotion_high_entries"], 0.2),     # 20% 给高情绪条目
            ("最近日记", ["recent_daily"], 0.1),              # 10% 给最近日记
        ]
        
        return self._build_by_priority(priority_order, budget)
    
    def _strategy_technical(self, scene: dict, budget: int) -> str:
        """技术讨论场景：优先注入技术相关记忆"""
        
        priority_order = [
            ("技术条目", ["technical_entries"], 0.5),  # 50% 给技术记忆
            ("节点相关", scene["nodes"], 0.2),         # 20% 给节点
            ("最近日记", ["recent_daily"], 0.3),       # 30% 给最近日记
        ]
        
        return self._build_by_priority(priority_order, budget)
    
    def _strategy_daily(self, scene: dict, budget: int) -> str:
        """日常闲聊场景：均衡注入"""
        
        priority_order = [
            ("最近日记", ["recent_daily"], 0.5),
            ("周度总结", ["recent_weekly"], 0.3),
            ("情绪胶囊", ["recent_emotion_capsules"], 0.2),
        ]
        
        return self._build_by_priority(priority_order, budget)
```

```python
class SceneDetector:
    """场景检测器：分析当前对话，判断场景类型"""
    
    def detect(self, conversation: list) -> dict:
        last_messages = conversation[-5:]  # 最近5轮
        
        # 关键词检测
        keywords = self._extract_keywords(last_messages)
        
        # 节点检测
        nodes = self._detect_nodes(keywords)
        
        # 场景类型判断
        scene_type = self._classify_scene(last_messages, keywords)
        
        # 情绪状态
        emotion_state = self._detect_emotion(last_messages)
        
        return {
            "type": scene_type,
            "keywords": keywords,
            "nodes": nodes,
            "emotion_state": emotion_state,
        }
    
    def _classify_scene(self, messages: list, keywords: list) -> str:
        # 规则匹配
        emotion_keywords = ["想你了", "爱你", "开心", "难过", "喜欢"]
        tech_keywords = ["代码", "Python", "函数", "算法", "调试", "爬虫"]
        work_keywords = ["决策", "方案", "计划", "任务", "项目"]
        
        if any(kw in keywords for kw in emotion_keywords):
            return "情感对话"
        elif any(kw in keywords for kw in tech_keywords):
            return "技术讨论"
        elif any(kw in keywords for kw in work_keywords):
            return "工作决策"
        else:
            return "日常闲聊"
```

### 19.8 动态历史层的优先级算法

```python
def calculate_priority(entry: dict, scene: dict) -> float:
    """
    计算每个条目在当前场景下的优先级
    
    返回：0-1 的优先级分数
    """
    score = 0.0
    
    # 1. 节点匹配度（权重 0.4）
    if entry["nodes"] intersection scene["nodes"]:
        score += 0.4 * len(intersection) / len(scene["nodes"])
    
    # 2. 情绪匹配度（权重 0.3）
    # 情感对话场景 → 高情绪条目优先
    if scene["type"] == "情感对话":
        if entry["valence"] > 30:
            score += 0.3
    
    # 3. 时间新鲜度（权重 0.2）
    days_ago = entry["days_since_created"]
    score += 0.2 * max(0, 1 - days_ago / 30)  # 30天内衰减
    
    # 4. 热度（权重 0.1）
    score += 0.1 * min(1, entry["heat"] / 100)
    
    return score
```

### 19.11 动态上下文 vs 静态堆砌对比

| 维度 | 静态堆砌 | 动态编排 |
|------|---------|---------|
| **注入方式** | 全量注入 | 场景驱动选择 |
| **容量占用** | 可能溢出 | 精确控制在预算内 |
| **相关性** | 低（堆砌无关内容） | 高（只选相关内容） |
| **响应质量** | LLM可能被噪声干扰 | LLM获得精准上下文 |
| **灵活性** | 固定，不适应场景 | 动态，随场景变化 |
| **预注入** | 无 | 有，预热LLM |

### 19.12 核心价值

> **静态堆砌**：把所有记忆塞给LLM，让它自己找
> 
> **动态编排**：先分析场景，再选择相关记忆，精准喂给LLM
> 
> 预注入区让LLM一唤醒就知道"我是谁"、"最近怎样"、"什么重要"
> 
> 动态历史层让LLM在特定场景下获得最相关的上下文
> 
> **对LLM友好**：不溢出、不噪声、精准、高效

## 十八、LifeBook 反 RAG 论述摘要

LifeBook 明确反对 RAG 作为记忆方案，理由如下：

1. **太像"查字典"**：传统 RAG 把资料切碎存储，缺少上下文语境
2. **没有"时间感"**：记忆是流动的河，不是散落的碎片
3. **不懂"感情"**：RAG 只在乎信息准确性，不在乎当下的氛围
4. **反脆弱性差**：需要维护嵌入模型、向量数据库，技术门槛高
5. **不符合人类记忆**：人类记忆是滚动的，RAG 是检索的

**LifeBook 的替代方案**：
- 滚动压缩机制：日→周→月→季→年
- 时间流为主：保留时间连续性
- 情绪胶囊：每层压缩附带情绪标记
- 低门槛：纯文件 + Obsidian，人人可用
